# Public failure matrix

The test references below are exact repository symbols. “Gap” is intentionally honest:
an adjacent unit test is not evidence of an end-to-end crash simulation.

| Scenario | Expected durable job/slot state | Current test evidence | Gap |
| --- | --- | --- | --- |
| Coordinator killed during launch | The dispatch claim remains durable, is reconciled to `dispatch_failed`, and cannot launch again until explicit retry. | `tests/entities/plan/test_plan_store.py::test_dispatch_reconciles_coordinator_killed_after_claim_before_launch`; `tests/app/runtime/test_orchestrator_runner.py::test_plan_dispatch_materializes_private_brief_and_starts_exactly_once`; `tests/features/agent_runner/test_job_reconciliation.py::test_reconcile_kills_verified_child_after_coordinator_death` | The test kills the exact coordinator process at the `PlanDispatcher` launch callback before any external runner starts. It does not simulate a crash after a runner has been spawned; that boundary is covered by the coordinator-child reconciliation drill. |
| Worker killed/interrupted during finalization | Job is terminal/partial or finalizing; checkpoint and inbox must be durable before cleanup; dirty slot remains quarantined until replay safely releases it. | `tests/features/agent_runner/test_job_reconciliation.py::test_reconcile_replays_exact_process_killed_after_checkpoint_before_cleanup`; `tests/features/agent_runner/test_job_reconciliation.py::test_reconcile_replays_crash_after_terminal_transition`; `tests/features/agent_runner/test_job_reconciliation.py::test_reconcile_reuses_checkpoint_after_crash_before_slot_release`; `tests/features/result_handoff/test_result_handoff_orchestrator.py::test_checkpoint_cleanup_failure_keeps_slot_dirty_and_review_ref_visible` | The drill terminates an exact helper after checkpoint/inbox durability and before cleanup, then proves fresh replay reuses the checkpoint and inbox row. It does not interrupt an in-flight Git subprocess, so filesystem-level Git-command races remain outside this bounded simulation. |
| PID reuse or durable identity mismatch | Reconciliation must not terminate the unrelated process; job and slot remain quarantined. | `tests/features/agent_runner/test_job_reconciliation.py::test_reconcile_does_not_terminate_live_process_with_mismatched_durable_identity`; `tests/features/agent_runner/test_process_identity.py::test_verified_termination_refuses_reused_pid_identity` | The drill uses a live process with deliberately mismatched durable identity. Exact OS PID reuse is not reliably forceable and is not claimed. |
| SQLite contention | Transaction is serialized or fails visibly; no duplicate dispatch/partial durable claim. | `tests/entities/plan/test_plan_store.py::test_cross_process_sqlite_lock_serializes_one_durable_dispatch_claim`; `tests/entities/plan/test_plan_store.py::test_ready_task_claim_is_atomic_across_two_plan_store_instances` | The lock holder and claimant are separate subprocesses synchronized with ready/release files; the test covers one bounded contention window, not arbitrary filesystem or database failure. |
| Workspace edited after checkpoint | Cleanup fails closed; checkpoint/ref and inbox remain visible; slot remains dirty. | `tests/features/result_handoff/test_slot_checkpoint.py::test_cleanup_fails_closed_when_another_process_edits_after_checkpoint`; `tests/features/result_handoff/test_result_handoff_orchestrator.py::test_checkpoint_cleanup_failure_keeps_slot_dirty_and_review_ref_visible` | The editor completes before cleanup is invoked; this proves the durable checkpoint is preserved on cross-process interference but not a simultaneous Git-command race. |
| Repeat dispatch/recovery after restart | Existing claim is not duplicated; failed dispatch is blocked until explicit retry; restart resumes from SQLite. | `tests/entities/plan/test_plan_store.py::test_cross_process_sqlite_lock_serializes_one_durable_dispatch_claim`; `tests/entities/plan/test_plan_store.py::test_dispatch_failure_is_durable_and_not_automatically_reclaimed`; `tests/app/runtime/test_plan_cli.py::test_cli_restart_requires_explicit_retry_after_failed_dispatch` | The CLI drill forces a launch failure before an external worker starts; it proves dispatch ownership and retry persistence, not recovery of a partially started external worker. |

Operational response: preserve the database, run `status`/`plan summary`, reconcile
without termination first, inspect the slot and inbox, then retry only after the durable
state explicitly permits it. Root acceptance remains a separate decision.
